Boot Flow

After power-up or hardware reset, the hardware will boot the KM4 core at 150MHz. The boot process is handled by the on-chip Boot ROM and is always executed by the KM4 core.

After executing the KM4 bootloader code, the KM4 will initialize the environment for the KM0 core.

  1. KM4 boots ROM

  2. KM4 secure boot (optional)

  3. KM4 boots to SRAM

  4. KM4 assists KM0 images loading and signature verification (optional)

The SoC boot flow is illustrated below.

The immutable ROM provides ISP (In-System Programming) service for:

  • Initial programming of blank devices

  • Erasing and reprogramming pre-programmed devices

When a rising edge is detected on the RESET pin, the system samples the level of trap pin (PB5):

  • LOW level: External hardware requests ISP service

  • HIGH level: Normal chip boot

The immutable ROM provides ISP (In-System Programming) service for:

  • Initial programming of blank devices

  • Erasing and reprogramming pre-programmed devices

When a rising edge is detected on the RESET pin, the system samples the level of trap pin (PB5):

  • LOW level: The external hardware request to start the ISP command handler will be ignored. If there is no request to execute the ISP command handler, a search for a valid user program will be conducted.

    • If a valid user program is found, control will be transferred to that program.

    • If no valid user program is found, a dead loop is initiated.

  • HIGH level: Normal chip boot

After executing the KM4 bootloader code, the KM4 will initialize the environment for the KR4 and DSP core.

  1. KM4 boots ROM

  2. KM4 secure boot (optional)

  3. KM4 boots to SRAM

  4. KM4 assists KR4 images loading and signature verification (optional)

  5. KM4 assists DSP images loading and signature verification (optional)

Boot from NOR or NAND Flash depends on the settings in OTP. Additionally, PSRAM or DDR can be selected for storing code and data.

Boot API

The boot API is used to obtain the cause of the chip boot, and the function prototype is:

u32 BOOT_Reason(void);

The default return value of this API is 0 when initially powered on, and return value of re-boot caused by other reasons can be found in the following table. Users can found macro-definitions about return value in the file sysreg_aon.h.

Items

Description

Introduction

Get boot reason

Parameters

None

Return

Boot reason. It can be any of the following values or combinations: - AON_BIT_RSTF_OCP: Enter deep-sleep mode when OCP happens - AON_BIT_RSTF_KM0_SYS: KM0 system reset - AON_BIT_RSTF_KM4_SYS: KM4 system reset - AON_BIT_RSTF_WDG0: KM0 watchdog reset - AON_BIT_RSTF_WDG1: KM4 secure watchdog reset - AON_BIT_RSTF_WDG2: KM4 non-secure watchdog reset - AON_BIT_RSTF_WARM_KM42PERI: KM4 warm reset - AON_BIT_RSTF_WARM_KM02PERI: KM0 warm reset - AON_BIT_RSTF_DSLP: Wakeup from deep-sleep mode - AON_BIT_RSTF_BOR: BOR Reset
