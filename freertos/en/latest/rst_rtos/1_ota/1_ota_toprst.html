Introduction

OTA (Over-The-Air) technology is a method for remotely updating device firmware through wireless networks. With OTA updates, devices receive new firmware via wireless network, eliminating the need for physical access to each device. This not only improves update efficiency, reduces maintenance costs and complexity, but also ensures devices receive timely security patches and performance optimizations, enhancing device performance and user experience.

As IoT device application scenarios become increasingly complex, we propose an enhanced OTA solution, which, in addition to supporting traditional wireless updates, introduces non-wireless transmission schemes. This enhancement is particularly suitable for devices with unstable network connections or without wireless connectivity. Our non-wireless transmission solution allows updates via wired connections (such as SPI, UART serial ports) or through storage media preloaded with update content (such as SD cards, file systems). This approach not only offers greater flexibility but also ensures device availability and data integrity in various environments.

Image Slot

Two image slots (OTA1 and OTA2) are allocated in Flash layout for redundancy and version control, as shown in the figure below. The boot system selects between these slots based on validation criteria.

Image Pattern and Version Number

Boot selection between OTA1/OTA2 mainly depends on the image pattern and version number in certificate and manifest. As shown in the figure below, there are 8-byte image pattern, 2-byte major and 2-byte minor version in certificate and manifest.

The image pattern is data with a fixed length of 8 bytes. The OTA selection process checks the validity of the image pattern, and the bootloader and application have different image pattern.

The version number is a 4-byte combination of the major version and the minor version. The OTA selection process compares the version number size, calculated as follows:

\[\text{Version number} = (\text{Major version} \ll 16) \, | \, \text{Minor version}\]

On the next boot after an OTA upgrade, the bootloader checks the validity of the image pattern and compares the version numbers to decide whether to boot from OTA1 or OTA2. Consistency between image pattern and versions ensures reliable boot. Each firmware must check the following items:

  1. Check the validity of the image pattern for both OTA1 and OTA2.

    • If only one image pattern is valid, boot from the valid image.

    • If both image patterns are invalid, boot fail.

  2. Compare the version number of OTA1 and OTA2.

    • If both OTA1 and OTA2 are valid and with different version numbers, the device will boot from the image with a bigger version.

    • If both OTA1 and OTA2 are valid but with the same version number, the device will boot from OTA1.

Note

  • Bootloader check image pattern and version numbers in manifest, and version number can be 0 to 32767.

  • Application check image pattern and version numbers in certificate, and version number can be 0 to 65535.

Anti-rollback

Anti-rollback is the function to prevent version rollback attack. When the anti-rollback is enabled, the version number in certificate or manifest must not be smaller than the anti-rollback version in OTP. Otherwise, this image will be regarded as invalid and the chip will not boot from invalid image. Normally, if OTA update is security-related, user can program a bigger anti-rollback version number in OTP and update image with a bigger major version at the same time to prevent rollback attack.

The anti-rollback flow is shown below. Once the anti-rollback is enabled, the device will compare the major version numbers abtained from OTA1 and OTA2 images respectively with the anti-rollback version number in OTP. If the major version number of the image is smaller than the anti-rollback version number, this image will be regarded as invalid.

OTA Compressed Image

When OTA Image Compression is enabled, the OTA image will be compressed and image size will be reduced, which can save the flash space effectively.

The OTA Image Compression only compresses the APP image which named with suffix of app.bin. After compression finishes, the compressed APP image will be generated in directory image, which named with suffix of app_compress.bin. And at the same time, the compressed APP image will be integrated into ota_all.bin.

The compression algorithm uses LZMA. LZMA stands for Lempel-Ziv-Markov chain Algorithm. It is a lossless data compression algorithm developed by Igor Pavlov, known for its high compression ratio and relatively low decompression memory requirements. The LZMA algorithm is commonly used in 7z format compressed files and is widely applied in 7-Zip software.

Here, we use compression rate to represent compression efficiency. The compression rate is computed as follows:

\[\text{compression rate (%)} = \left( \frac{\text{compressed image size}}{\text{original image size}} \right) \times 100\]

The compression rate depends on the image content. Usually, the compression rate is in the range of 60 to 70 percent. The more complex of the original image content, the higher compression rate will be.

Once the compressed APP image is download into one OTA slot, the system will always select the compressed image first. The compressed image will be decompressed into another OTA slot, and then the system will boot from this slot. This means there is always only one valid APP image between the two slots. For example, if the board is currently booting from OTA1 and running the OTA application, the compressed image will be downloaded into OTA2. On the next boot, the system will select the compressed image, decompress it into OTA1, and then continue to boot from OTA1.

Users can generate OTA compressed image by the following steps:

  1. Update the version number in configuration file manifest.json located in project folder.

    1. Configure major and minor version for KM4 bootloader.

      "boot":
{
   "IMG_ID": "0",
   "IMG_VER_MAJOR": 1,
   "IMG_VER_MINOR": 1,

    2. Configure major and minor version for certificate and application.

      "app":
{
   "IMG_ID": "1",
   "IMG_VER_MAJOR": 1,
   "IMG_VER_MINOR": 1,
   ...
},

  2. Adjust the bootloader version of anti-rollback and enable anti-rollback if necessary.

    1. Change the bootloader version of anti-rollback

      By default, all images use the same anti-rollback version in OTP as threshold to prevent anti-rollback attack.

      Name

      OTP address

      Length

      Description

      BOOTLOADER_VERSION

      Physical 0x36E~0x36F

      16 bits

      Bootloader version of anti-rollback

      The bootloader version of anti-rollback is 0 by default. Users can change the number of ‘0’ bit to enlarge the bootloader version.

      For example, users can program the bootloader version of anti-rollback to 1 by the following command:

      EFUSE wraw 36E 2 FFFE

    2. Enable anti-rollback

      Users can program OTP by the following command to enable anti-rollback.

      EFUSE wraw 368 1 BF

    Note

    • Once anti-rollback is enabled, it cannot be disabled.

    • If bootloader and application do not use the same anti-rollback version, modify BOOT_OTA_GetCertRollbackVer() in {SDK}\component\soc\amebaxxx\bootloader\boot_ota_km4.c or {SDK}\component\soc\amebaxxx\bootloader\boot_ota_hp.c and define another anti-rollback version in OTP for the application.

  3. Write the bootloader OTA2 address into OTP if users need to upgrade the bootloader. Set this address according to Flash_Layout refer to User Configuration.

    EFUSE wraw 36C 2 0082

    EFUSE wraw 36C 2 0082

    EFUSE wraw 36C 2 0082

    EFUSE wraw 36C 2 0083

    Note

    • The bootloader OTA2 address is the value of OTP 0x36C with 12-bit left shifted, or is the value of OTP 0x36C * 4K.

    • If the default address of bootloader OTA2 is 0xFFFFFFFF, the bootloader upgrade won’t occur during OTA, and the board always boots from bootloader OTA1.

    • Use commands above in the serial terminal tool.

  4. Rebuild the project.

  5. Download the images into Flash, and reset the board.

    [MODULE_BOOT-LEVEL_INFO]: IMG2 BOOT from OTA 1

OTA Flow

The image upgrade is implemented in the following steps:

  1. Connect to the server. The IP address, port, and OTA type are needed for OTA via network.

  2. Determine the target slot to be upgraded. If the current running slot is OTA1, the OTA2 would be selected to upgrade. Otherwise, the OTA1 would be selected.

  3. Receive the OTA header to obtain information about the target upgrade firmware, as shown in the table above.

  4. Download the new firmware from the server.

  5. Erase the flash space for new firmware and write it into flash except for the manifest data.

  6. Verify the checksum:

    • If the calculated checksum doesn’t match the checksum obtained from OTA header, the OTA process fails.

    • If the calculated checksum maches the checksum obtained from OTA header, write the manifest data into the upgraded region to indicate the board should boot from the new firmware next time.

  7. OTA is finished. Reset the board, and it would boot from the new firmware.

Note

The manifest data mentioned above includes image pattern and version numbers. The image pattern and version consistency ensure a reliable boot

OTA Demo

Follow these steps to run the OTA demo to upgrade from HTTP server:

  1. Edit example_ota.c.

    1. Edit the host according to the server IP address.

      #define PORT   8082
static const char *host = "192.168.31.193";   //"m-apps.oss-cn-shenzhen.aliyuncs.com"
static const char *resource = "ota_all.bin"; //"051103061600.bin"

    2. Edit the OTA type to OTA_HTTP.

      ret = ota_update_init(ctx, (char *)host, PORT, (char *)resource, OTA_HTTP);

  1. Rebuild the project with the command ./build.py -a ota and download the images to the device.

  2. Modify the major and minor version number in manifest.json with a bigger version as described in Version Number.

    Note

    The bootloader will select OTA image with a bigger version number by default. If users don’t want to modify the version number, modify OTA_CLEAR_PATTERN to 1 defined in ameba_ota.h before Step 2. It should only be used in the development stage.

  3. Rebuild the project and copy ota_all.bin into DownloadServer(HTTP).

  4. Edit http_server.py.

    server_ip = '192.168.31.193'
server_port = 8082

  5. Execute the script by command python http_server.py, and start the download server program.

  6. Reset the board and connect to the HTTP server.

  7. After image download finished, the board will reset automatically and boot from the new firmware .
