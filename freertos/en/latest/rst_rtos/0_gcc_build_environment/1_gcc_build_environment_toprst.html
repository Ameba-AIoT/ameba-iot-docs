Introduction

This chapter illustrates how to build Realtek’s SDK under GCC environment. It focuses on both Windows platform and Linux distribution. The build and download procedures are quite similar between Windows and Linux operating systems.

  • For Windows, Windows 10 64-bit is used as a platform.

  • For Linux server, Ubuntu 22.04 64-bit is used as a platform.

The GCC environment includes the CMake compilation environment, Python environment, and other tool software, as shown in the table below。

Users can set up the environment in two ways:

  1. Download the bundled software package we provide and run a script to automatically configure the environment variables before compiling. This method does not affect the user’s existing software environment and avoids compatibility issues caused by software versions.

  2. Manually download the required tool software and configure the corresponding environment variables. If users wish to use their global environment, they can also manually download the software, but this approach might impact other environments of the user.

  • Download the compressed package and extract it to the /opt/rtk-toolchain folder.

mkdir /opt/rtk-toolchain
cd /opt/rtk-toolchain
wget https://github.com/Ameba-AIoT/ameba-toolchain/releases/download/prebuilds-v1.0.0/prebuilts-linux-1.0.0.tar.gz
tar -xzf prebuilts-linux-1.0.0.tar.gz

Note

  • The compressed package needs to be extracted to the specified /opt/rtk-toolchain path. If need to change the path, you must also modify the variables in ameba.sh.

  • If you encounter network error during downloading, please use the aliyun URL: https://rs-wn.oss-cn-shanghai.aliyuncs.com/prebuilts-linux-1.0.0.tar.gz

  • Since a portable version of Python cannot be provided under Linux, users need to install Python manually.

sudo apt install python3 python3-pip python3-venv

Note

  • Run python --version to check version of python, recommanded to use version greater than 3.10.

  • If there are multiple versions of Python on the host, you can use the command update-alternatives --install /usr/bin/python python /usr/bin/python3.x 1 to select a specific version of Python, where x in python3.x 1 represents the desired version number.

  • If you encounter the error command 'python' not found, try running the following command to solve the issue: ln -s /usr/bin/python3 /usr/bin/python

  • Enter into the root directory of SDK and run source ameba.sh, which will automatically configure the environment variables.

source ameba.sh

Note

The environment variables will be reset after terminal closed. Therefore, the scripts need to be run again when you open a new terminal.

  • Create a new folder C:\rtk-toolchain under Disk C.

  • Download the compressed package and extract it to the C:\rtk-toolchain folder. You can click on any of the links below to download. Users in mainland China are advised to use the aliyun link.

    Download from github

    Download from aliyun

    Note

    The compressed package needs to be extracted to the specified C:\rtk-toolchain path. If you need to change the path, you must also modify the relevant variables in the ameba.bat script.

  • Navigate to the SDK root directory and double-click the ameba.bat script. This will automatically configure the environment variables, then you can type commands in this opened terminal.

    Note

    If users want to use a Unix-like command-line environment such as Git Bash or MSYS2 on Windows, please open a new terminal in the SDK root directory and run source ameba.sh.

  • This method relies on Python virtual environment. If you encounter Python system errors, such as messages indicating that a DLL dynamic library missing, try removing the .venv folder in the root directory of the SDK and then run the ameba.bat script again. This will automatically reinstall the Python virtual environment.

  1. This step requires users to have sudo permissions:

    sudo apt-get install cmake ninja ccache wget python3 python3-pip python3-venv

  2. Install pip packages:

    Enter into the root directory of SDK and run：

    pip install -r tools/requirements.txt

    Note

    For the installation of pip packages, it is recommended that customers use a Python virtual environment. This approach avoids interference from Realtek’s pip packages with the user’s existing local packages.

    1. Enter into the root directory of SDK and run python -m venv .venv to create a virtual environment.

    2. Run source .venv/bin/activate to activate the virtual environment. (This activation command needs to be run each time the terminal is restarted.)

    3. With the virtual environment activated, run pip install -r tools/requirements.txt. The required packages will be installed under the .venv directory.

Chocolatey is a package manager for Windows, which can manage the complete lifecycle of software packages. If you can run PowerShell with administrator privileges, using Chocolatey to install packages is more convenient.

  1. Install Chocolatey

    First, run Windows PowerShell as administrator. If you have correct privilege, the title of opening terminal will be “Administrator: Windows PowerShell”.

    Execute the following two commands in Administrator mode：

    Set-ExecutionPolicy AllSigned

Set-ExecutionPolicy Bypass -Scope Process -Force; [System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072; iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))

  2. Install software:

    choco feature enable -n allowGlobalConfirmation

choco install cmake --installargs 'ADD_CMAKE_TO_PATH=System'

choco install ninja ccache python310 wget 7zip

  3. Install pip packages:

    Enter into the root directory of SDK and run:

    pip install -r tools\requirements.txt

    Note

    For the installation of pip packages, it is recommended that customers use a Python virtual environment. This approach avoids interference from Realtek’s pip packages with the user’s existing local packages.

    1. Enter into the root directory of SDK and run python -m venv .venv to create a virtual environment.

    2. Run .venv\Scipts\activate.bat to activate the virtual environment. (This activation command needs to be run each time the terminal is restarted..)

    3. With the virtual environment activated, run pip install -r tools\requirements.txt. The required packages will be installed under the .venv directory.

Installing Toolchain

By default, the toolchain will be automatically installed in default path when building the project at the first time:

  • Linux：/opt/rtk-toolchain

  • Windows：C:\rtk-toolchain

  1. Before the compilation, the build scripts will check if the toolchain exists and if the version of the toolchain match the lastest version. Once error occurs, you should fix the error according to the prompts on the screen and try building again.

  1. The toolchain compressed package is available on GitHub, and the system will attempt to download the toolchain from GitHub by default before the first build. If you encounter any issues with the download, please verify that your host machine can correctly access GitHub. If you cannot access GitHub, please use the following commands to switch the download source:

    build.py -D USE_ALIYUN_URL=True

  1. If you need to modify the toolchain installation path, please update the TOOLCHAINDIR variable in the cmake/toolchain/ameba-toolchain-xxx.cmake file.

Note

  • If an error Create Toolchain Dir Failed. May Not Have Permission appears, please create the installation directory by manual. If still fails, refer to 3 above to change the installtion path.

  • If an error Download Failed appears, please check if the network connection is accessible first. If still fails, refer to 2 above to intall the toolchain again.

  • If an error Current Toolchain Version Mismatched appears, please delete the current toochain and retry with build.py, and the latest toolchain will be installed automatically during building the project.

If the installation still fails, try with the manual installation steps.

  1. Create a rtk-toolchain folder on drive C.

  2. Download the toolchain compressed package.

    Please refer to the current SDK’s toolchain version and retrieve the corresponding toolchain compressed package URL from either ${SDK}/cmake/toolchain/ameba-toolchain-asdk-${v_ASDK_VER}.cmake or ${SDK}/cmake/toolchain/ameba-toolchain-vsdk-${v_VSDK_VER}.cmake files. Here, v_ASDK_VER and v_VSDK_VER represent the respective toolchain versions, which can be found in the ${SDK}/CMakeLists.txt file.

  3. Extract the asdk-10.3.x-mingw32-newlib-build-xxxx.zip to C:\rtk-toolchain and rename it to asdk-10.3.x-xxxx. Here, xxxx represents the numeric version of the toolchain. Including the version number in the extracted folder name prevents overwriting different versions of the software.

  1. Create a folder named rtk-toolchain under the /opt directory.

    mkdir /opt/rtk-toolchain

  2. Download the toolchain compressed package to /opt/rtk-toolchain.

    Refer to the current SDK’s toolchain version and retrieve the corresponding toolchain compressed package URL from either ${SDK}/cmake/toolchain/ameba-toolchain-asdk-${v_ASDK_VER}.cmake or ${SDK}/cmake/toolchain/ameba-toolchain-vsdk-${v_VSDK_VER}.cmake files. Here, v_ASDK_VER and v_VSDK_VER represent the respective toolchain versions, which can be found in the ${SDK}/CMakeLists.txt file.

  3. Extract asdk-10.3.1-linux-newlib-build-xxxx.tar.bz2 to /opt/rtk-toolchain and rename it to asdk-10.3.x-xxxx. Here, xxxx represents the numeric version of the toolchain. Including the version number in the extracted folder name prevents overwriting different versions of the software. For example, after extracting a compressed package named asdk-10.3.1-linux-newlib-build-4291-x86_64_with_small_reent.tar.bz2, the extracted floader should be renamed to asdk-10.3.1-4291.

Note

  • The extraction path for the toolchain must match the default path, unless you modify the default toolchain path in cmake/toolchain/ameba-toolchain-xxx.cmake.

  • If no available toolchain is detected under the path defined in cmake/toolchain/ameba-toolchain-xxx.cmake, a download task will be create automatically when building the project at the first time.

  • If a correct version of the compressed package is detected in the path defined in cmake/toolchain/ameba-toolchain-xxx.cmake, the script will also automatically perform the extraction and renaming operations.

Configuring SDK (menuconfig)

This section illustrates how to change SDK configurations with menuconfig.py.

Configuration Steps

  1. Switch to the {SDK}\amebadplus_gcc_project directory.

  2. Run the menuconfig.py command in a Windows Command Prompt window or a Linux terminal.

Important Compilation Configuration Options

  • General Config: Shared configuration options for KM4 and KM0, these settings will affect both KM4 and KM0.

  • Network Config: Shared configuration options for KM4 and KM0, these settings will affect both KM4 and KM0.

  • KM4 Config: Configuration options specific to KM4, these settings will only affect KM4 and not KM0.

  • KM0 Config: Configuration options specific to KM0, these settings will only affect KM0 and not KM4.

We provide a build.py script to simplify the build command of projects. Common commands are listed below:

Function

Command

Description

Help

build.py -h

List supported commands

Build

build.py

Incremental build of project

Pristine Build

build.py -p

Remove all products and build

Specify Example

build.py -a <APP>

Build example named ‘APP’

Specify Target

build.py -g target

Build specified target

Cleanup

build.py -c

Clean compilation products

Download

build.py -gdb

Download firmware via GDB

Debug

build.py -debug

Enter debug mode

  1. Help command

    If you are using the build.py script for the first time, use the following command to understand its usage.

    build.py -h

    With the -h or --help option, all supported commands will be listed.

  2. Pristine Build

    build.py -p

    Pristine build means that there are no previous residual products affecting this build. All previously products will be removed before this build. The -p option is equivalent to the --pristine option.

  3. Incremental Build

    build.py

    Without any arguments, the project will undergo incremental build, which significantly saves time compared to full build.

  4. Specify Build Directory and Build

    build.py -d <BUILD_DIR>

    This command allows specifying the directory where intermediate products will be stored. If not specified, it defaults to creating a build folder.

  5. Build Specified Example

    build.py -a <APP>

    A rich set of application examples are provided under the ${SDK}/component/example directory. To build one of these examples, pass the application name to build.py using the -a or --app options.

    For instance, to build the ${SDK}/component/example/network_protocol/http_client example, you can input the following command:

    build.py -a http_client

    If you do not pass an app name through -a, it will build a default empty project.

  6. Build Specified Target

    build.py -g <TARGET>

    By default, all necessary targets will be built together. Specific targets can be built individually using the -g or --target options. Use the -h option to view all available targets.

    For example, to regenerate imgtool flashloader, enter the following command:

    build.py -g gen_imgtool_floader

  7. Project Cleanup

    build.py -c

    Both -c and --clean clean up the current project’s products and intermediate files.

  8. Pass CMake Cache Variables

    build.py -D CACHE_VAL1=VALUE1 CACHE_VAL2=VALUE2 …

    CMake cache variables’ initial values can be set using the -D option. To inherit this functionality, build.py also supports the -D option. Predefined values by CMake and user-defined values can both follow after -D. More details about cache variables for this project can be found in the CMakeLists.txt file under the ${SDK}/amebaxxx_gcc_project directory.

    Note

    • Native CMake command requires each variable definition to have a -D prefix, e.g., -DCACHE_VAL1=VALUE1 -DCACHE_VAL2=VALUE2

    • However, build.py requires all variables to follow a single -D with spaces, e.g., -D CACHE_VAL1=VALUE1 CACHE_VAL2=VALUE2
