Introduction

  1. IPC Definition

    • Inter-Processor Communication (IPC) is a hardware-accelerated communication mechanism designed for multi-core heterogeneous systems. It enables low-latency cross-core data exchange through shared SRAM, ensuring efficient collaboration between KM0 and KM4 processors.

  1. Hardware Architecture Features

    • Symmetric Structure

      • Each CPU is equipped with identical IPC modules.

    • Full-Duplex Communication Model

      • Supports bidirectional full-duplex communication: 16 independent channels per direction (KM0↔KM4)

      • 32 dedicated physical links in total

    • Channel Isolation

      • Hardware-level resource isolation ensures non-blocking parallel operations across channels

      • Enables concurrent multi-channel transmissions (e.g., KM0 can send sensor data via Channel n while transmitting logs via Channel m simultaneously)

    • Address Mapping Mechanism

      • Shared SRAM region: A physically contiguous memory block with unified address mapping, providing coherent memory access across cores

      • Automatic interrupt triggering upon write operations enables hard real-time response

    • Zero-Copy Transmission

      • Pointer-based mode: Direct address sharing eliminates memory copy overhead

  2. IPC Operational Mechanism

    • Key Registers (KM0-side example)

      • Data Channels

        • REG_TX_DATA_L2N

        • REG_RX_DATA_N2L

      • Interrupt Status Register

        • REG_ISR_KM0

      • Interrupt Mask Register

        • REG_IMR_KM0

      • Interrupt Clear Register

        • REG_ICR_KM0

    • End-to-End Trigger Mechanism (KM0→KM4 transmission example)

      1. When bit ‘1’ is written to channel n (n=0,1,…15) in REG_TX_DATA_L2N:

        • Corresponding bit in KM4’s REG_RX_DATA_L2N is set

        • Rx Full interrupt bit for channel n in REG_ISR_KM4 is triggered

      2. Clearing channel n’s Rx Full interrupt in REG_ISR_KM4:

        • Automatically sets Tx Empty interrupt bit for channel n in REG_ISR_KM0

        • Clears corresponding bits in both REG_TX_DATA_L2N and REG_RX_DATA_L2N

  3. Data Flow Model

    • Transmitter-Side Flow：

      [Application Data] → Acquire Available Channel → Write to Shared Memory → Update Channel Register → Trigger Peer Interrupt → Peer Reads Memory

    • Receiver-Side Flow：

      Interrupt Notification → Read Channel Register → Locate Shared Memory → Process Data → Clear Interrupt Flag

  1. IPC Definition

    • Inter-Processor Communication (IPC) is a hardware-accelerated communication mechanism designed for multi-core heterogeneous systems. It enables low-latency cross-core data exchange through shared SRAM, ensuring efficient collaboration between KM0, AP and NP processors.

How to Use IPC

  1. Channel Selection and Definition

    • Objective: Enable channel n for KM0→KM4 communication

    • File Path : ameba_ipccfg.h

    • Configuration Steps：

      1/* Uncomment channel n and add functional description */
2#define IPC_N2A_Channeln  n  /*!< User-defined channel: KM0→KM4 sensor data transmission  */
    • Important Notes：

      • Avoid using system-reserved channels (some channels are preallocated for WiFi/BT systems)

      • Channel number must be within 0-15 range

  2. Communication Protocol Stack Registration

    • IPC Configuration Table Parameters

      <USER_MSG_TYPE>:

      Specifies data transfer mode

      • IPC_USER_DATA: Direct 32-bit value transfer

      • IPC_USER_POINT: Pointer transfer for data >32 bits

      <RxFunc>:

      The callback function that CPU executes after receiving IPC.

      <RxIrqData>:

      The parameters that are passed into the rx callback function.

      <TxFunc>:

      The callback function that CPU executes after sending IPC.

      <TxIrqData>:

      The parameters that are passed into the tx callback function

      <IPC_Direction>:

      Communication direction

      • IPC_KM4_TO_KM0: KM4→KM0 transmission

      • IPC_KM0_TO_KM4: KM0→KM4 transmission

      <IPC_Channel>:

      Physical channel number (0-15)

    • IPC Data Structure

      1typedef struct ipc_msg_struct {
2   u32 msg_type;   //Same as USER_MSG_TYPE
3   u32 msg;        //32-bit data payload
4   u32 msg_len;    //Data length (bytes)
5   u32 rsvd;
6} IPC_MSG_STRUCT, *PIPC_MSG_STRUCT, ipc_msg_struct_t;

    • Interrupt Callback Registration (KM4 Receiver)

      • Add IPC_INIT_TABLE for KM0→KM4 channel n in KM4:

       1/* Channel configuration table */
 2IPC_TABLE_DATA_SECTION
 3const IPC_INIT_TABLE ipc_chn_config = {
 4   .msg_type    = IPC_USER_POINT,     // Pointer-based transmission
 5   .rx_func     = KM4_Channeln_Handler,
 6   .rx_irq_data = NULL,               // No additional parameters
 7   .tx_func     = IPC_TXHandler,      // SDK-standard TX completion callback
 8   .tx_irq_data = NULL,
 9   .direction   = IPC_KM0_TO_KM4,     // Direction identifier
10   .channel_num = IPC_N2A_Channeln    // Channel macro
11};

      • The SDK automatically enables IPC interrupts and registers IRQ handlers. When KM0 transmits via channel n, KM4 executes KM4_Channeln_Handler(). The SDK clears channel ISR flags upon IRQ exit.

      1/* KM4 receive handler (interrupt context) */
2void KM4_Channeln_Handler(void *data) {
3   PIPC_MSG_STRUCT msg = ipc_get_message(IPC_KM0_TO_KM4, IPC_N2A_Channeln);
4   uint32_t *sensor_data = (uint32_t*)msg->msg;  // Extract shared memory data
5   _memcpy(data, (void*)sensor_data, sizeof(data));  // Memory barrier protection
6   process_sensor_data(data);            // User data processing
7}
    • Transmitter Configuration (KM0 Side) -Optional

      • For non-blocking bulk transfers, register TX IRQ handler using IPC_TXHandler() from ameba_ipc_api.c . This releases channel semaphore (tx_sema) to enable continuous transmission.

       1/* TX completion callback */
 2void IPC_TXHandler(uint32_t status) {
 3
 4   IPC_INTConfig(IPCx, ChanNum, DISABLE); // Disable interrupt
 5   ...
 6   rtos_sema_give(tx_sema);  // Release semaphore for next transfer
 7   ...
 8
 9}
10
11/* KM0 TX configuration table */
12IPC_TABLE_DATA_SECTION
13const IPC_INIT_TABLE ipc_km0_tx_config = {
14   .msg_type    = IPC_USER_POINT,
15   .rx_func     = NULL,              // No receive function
16   .tx_func     = IPC_TXHandler,    // SDK-standard/custom TX callback
17   .tx_irq_data = NULL,
18   .direction   = IPC_KM0_TO_KM4,     // Direction identifier
19   .channel_num = IPC_N2A_Channeln    // Channel macro
20};

  3. Data Transmission Implementation

    • KM0 Transmission

      • Call ipc_send_message() with channel number and message. Set third parameter to NULL if no payload.

       1void send_sensor_data(void) {
 2
 3   IPC_MSG_STRUCT ipc_msg_temp;
 4   sensor_buffer = read_sensor();  // Update shared memory
 5
 6   ipc_msg_temp.msg_type = IPC_USER_POINT; // Pointer transfer
 7   ipc_msg_temp.msg = (u32)&sensor_buffer;
 8   ipc_msg_temp.msg_len = 1;              // Data length (bytes)
 9   ipc_msg_temp.rsvd =0;
10
11   /* Send with timeout handling */
12   ipc_send_message(IPC_KM0_TO_KM4, IPC_N2A_Channeln, & ipc_msg_temp);
13}
    • KM4 Reception & Task Coordination

      • Upon IPC reception, KM4 enters interrupt context and triggers task-level processing via semaphore.

       1   /* Interrupt context: Fast data capture */
 2   void KM4_Channeln_Handler(void *data) {
 3
 4      // get message
 5
 6      rtos_sema_give(ipc_sema);  // Trigger task-level processing
 7   }
 8
 9/* Task context: Data processing */
10   void ipc_task(void *param) {
11      while (1) {
12         rtos_sema_take(ipc_sema, RTOS_WAIT_FOREVER);
13         data_pipeline_process(data);
14      }
15   }

  2. Communication Protocol Stack Registration

    • IPC Configuration Table Parameters

      <USER_MSG_TYPE>:

      Specifies data transfer mode

      • IPC_USER_DATA: Direct 32-bit value transfer

      • IPC_USER_POINT: Pointer transfer for data >32 bits

      <RxFunc>:

      The callback function that CPU executes after receiving IPC.

      <RxIrqData>:

      The parameters that are passed into the rx callback function.

      <TxFunc>:

      The callback function that CPU executes after sending IPC.

      <TxIrqData>:

      The parameters that are passed into the tx callback function

      <IPC_Direction>:

      Communication direction

      • IPC_LP_TO_NP: LP→NP transmission

      • IPC_NP_TO_LP: NP→LP transmission

      • IPC_LP_TO_AP: LP→AP transmission

      • IPC_AP_TO_LP: AP→LP transmission

      • IPC_AP_TO_NP: AP→NP transmission

      • IPC_NP_TO_AP: NP→AP transmission

      <IPC_Channel>:

      Physical channel number (0-15)

Troubleshooting

Phenomenon

Cause

Solution

Channel conflict

Two IRQs (interrupt requests) have been registered for the same channel

Check if the channel is registered twice

IPC send timeout

  1. Multiple threads continuously send IPC through the same channel

Register interrupt handler for tx empty, yield scheduling by waiting for os semaphore when channel is busy, to avoid deadlock

  1. Rx side receives data too slowly

Check if the RX side is functioning properly

IPC semaphore acquisition timeout

  1. Multiple threads continuously send IPC through the same channel

Avoid having more than two threads use the same channel simultaneously

  1. Rx side receives data too slowly

Check if the RX side is functioning properly

Rx side no response

  1. Interrupt on Rx side is not enabled

Use InterruptRegister to enable interrupt for IPC, call IPC_INTConfig, and enable the corresponding channel interrupt.

  1. Channel direction is configured incorrectly

Check the direction field in IPC_INIT_TABLE

IPC Application

Introduction

IPC queue is designed for cores communication. Define an IPC queue between every two cores, and the two cores can send and receive messages through this IPC queue.

IPC Message Queue:

  1. Packets send data from Core A to Core B through IPC message queue.

  2. Sender copy a complete packet into queue.

  3. Receiver copy packet with custom size from queue.

For receiver, IPC interrupt is necessary.

How to Use IPC Application

Before use IPC application, users should enable this function and initial it in each core.

  • For KM4: make menuconfig > MENUCONFIG FOR KM4 CONFIG > CONFIG IPC Message Queue > Enable IPC Message Queue

  • For KR4: make menuconfig > MENUCONFIG FOR KR4 CONFIG > CONFIG IPC Message Queue > Enable IPC Message Queue

  • For DSP: Open project_dsp with Xplorer, then click project_dsp > Build Properties > Symbols, and set CONFIG_RPC_EN to 1

The procedure will call ipc_app_init() in main() to initial IPC application when IPC application is enabled. The initialization include IPC queue initialization.

How to Use IPC Queue

For example, communication between KM4 and KR4 through IPC_ID_KR4_TO_KM4.

  1. IPC_ID_KR4_TO_KM4 definition in enumeration IPC_ID in ipc.h is shown below. This definition is used in Stap 2.

    typedef enum {
   IPC_ID_KR4_TO_KM4 = 0,
   IPC_ID_KM4_TO_KR4 = 1,
   IPC_ID_KR4_TO_DSP = 2,
   IPC_ID_DSP_TO_KR4 = 3,
   IPC_ID_DSP_TO_KM4 = 4,
   IPC_ID_KM4_TO_DSP = 5,
   /*14-19 used for RPC*/
   IPC_ID_NUM        = 20
} IPC_ID;

  1. When KR4 sends a message to KM4 through IPC_ID_KR4_TO_KM4, it should call IPC_Message_Queue_Send() and specify the queue ID and message.

    int32_t ret =  IPC_Message_Queue_Send(IPC_ID_KR4_TO_KM4, buf, IMQ_BUF_SIZE, WAIT_FOREVER);

  2. When KM4 receives a message from KR4 through IPC_ID_KR4_TO_KM4, it should call IPC_Message_Queue_Recv() and the queue ID and a buffer to receive message.

    size = IMQ_BUF_SIZE;
int32_t ret =  IPC_Message_Queue_Recv(IPC_ID_KR4_TO_KM4, buf, &size);

Note

  • The memory regions are only defined in AP core, the maximum data region is 40kB.

  • Users can use default test demo to verify IPC queue by command test_ipc_message in console both in KR4 and KM4.
