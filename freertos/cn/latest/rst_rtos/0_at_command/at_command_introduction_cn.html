概述

本节描述AT指令的作用、使用方法和版本信息。

AT指令模式

当前支持的AT指令模式分为两种：LOGUART模式与MCU控制模式，分别适合不同的开发场景。

LOGUART模式

  • 适用场景：模块功能评估与测试验证

  • 典型应用：Wi-Fi/BT射频性能测试

备注

此模式下，AT指令与驱动日志都是通过LOGUART输出的，因此AT指令信息可能会被日志中断，建议在非量产阶段使用。

MCU控制模式

  • 适用场景：客户产品量产开发

  • 应用说明：AT指令通过UART/SPI/SDIO通信接口独立传输和显示，无日志干扰

Developing

我们将Ameba模块设置为从机，MCU作为主机。主机可以向从机发送AT命令，并接收相应的AT响应。

AT命令包括多种类型：如Wi-Fi AT命令、MQTT AT命令、WebSocket AT命令、TCP/IP AT命令和蓝牙AT命令，以支持多样化的通信需求。

硬件连接

在进行配置前，需准备以下硬件：

LOGUART模式

  • Ameba板：作为从机模块

  • PC（或其他主机设备） ：用于输入AT指令，观察AT命令的响应

  • LOGUART：连接模块与PC（或其他主机设备），用于下载固件、传输驱动日志和AT命令日志

在LOGUART模式下，AT命令的输入和响应显示在同一端口。

MCU控制模式

  • Ameba板：作为从机模块

  • 树莓派（或其他MCU主机设备）：用于输入AT命令，观察AT命令的响应。

  • LOGUART：将模块连接到树莓派（或其他MCU主机设备），用于下载固件和显示驱动日志

  • SPI/UART/SDIO：将模块连接到树莓派（或其他MCU主机设备），用于传输AT命令和显示AT命令响应

在MCU控制模式下，AT命令的输入和响应可以与驱动日志分离开来，用户能更直观地查看AT命令的执行结果。

在MCU控制模式下，用户需要提前准备 atcmd_config.json 文件（参考 配置）。

如果未提供VFS AT命令配置文件，将使用UART的默认配置。

命令描述

命令格式

当前支持的AT命令集的格式以两个大写字母 AT （Attention的缩写）开头，称为起始字符；后接一个 + 号；之后是命令名称。如果有多个参数，则在命令名称后接一个 =，再接参数列表。

示例

AT+COMMAND=parameter1,parameter2

在这个示例中：

  • AT：起始字符，表示当前字符串可以被识别为AT命令

  • +：用于分隔起始字符和后续命令

  • COMMAND：具体的命令名称，需要立即执行

  • 此命令需要两个参数： parameter1 和 parameter2

某些情况下，在AT命令中可以省略某些参数，此时参数之间应输入一个或多个逗号。

示例

AT+COMMAND=parameter1, ,parameter3

在这个AT命令中，两个逗号之间存在一个被省略的 parameter2，此时使用默认值。

命令响应

在接收到AT命令后，从机首先判断其是否为有效命令。如果被视为无效命令（不在AT命令集内），将显示 unknown command COMMAND。否则，将根据输入的命令及参数执行相应操作。

  • 当命令执行成功时，通常会返回命令名称加上 OK 标记。

  • 当命令执行失败时，通常会返回命令名称加上 ERROR 标记，并附加一个错误代码。

备注

每个AT命令都有其对应的错误代码编号和意义。

命令参数

当介绍AT命令的参数列表时，遵循以下约定：

  • 尖括号< >表示参数名

  • 方括号[ ]表示该参数为可选参数

  • 不同的参数之间用逗号分隔

示例

AT+COMMAND=<param1>[,<param2>,<param3>]

在这个AT命令中，第一个参数 param1 是必须的，第二个参数 param2 和第三个参数 param3 是可选的。

转义字符

在某些AT命令中，如果确实需要将逗号作为参数的一部分，建议使用转义字符 \。反斜杠本身用转义字符表示为 \\。

示例

AT+COMMAND=parameter1,head\,tail,head\\tail

在这个AT命令中，总共有三个参数：

  • 第二个参数 head,tail 是一个包含逗号的字符串。 head,tail 中的逗号不会被视为参数的分隔符，而是字符串的一部分。

  • 第三个参数 head\tail 是一个包含反斜杠的字符串。单个反斜杠是非法的，换句话说，在AT命令中，单个反斜杠后面必须跟随一个逗号或另一个反斜杠。

对于其他不需要使用转义字符的AT命令，逗号通常被视为分隔符，单个反斜杠可作为普通字符使用。

命令长度

每个AT命令必须不超过指定的长度限制，否则，多余的部分将被忽略。

长度限制有两种类型：

  • 当启用了较长的命令格式时，长度限制为4095字节。

  • 当使用较短的命令格式时，长度限制为126字节。

在使用转义字符的AT命令中，转义字符（如 \ 或 \\）应被视为占用2个字节。

在编译SDK时，可通过 make menuconfig 命令来修改AT命令长度限制。如果选择了 Enable Longer CMD，则命令长度限制将变大（即4095字节）。

透传模式

在透传传输模式下，通过命令传输的数据长度仅受到可用堆内存大小的限制。传输的数据不受格式检查的约束，也不会因命令结束字符（ \r\n）而终止。

在执行支持透传模式的命令后，AT模块将返回 >>> 指示已进入透传模式。此时，MCU需要传输AT命令中指定长度的数据。一旦接收到指定长度的数据，AT模块将自动退出透传模式，并返回AT命令执行的结果。

备注

仅MCU控制模式支持透传模式。

AT命令版本

用户可以通过执行 AT+GMR 命令查询当前固件的AT命令版本信息。

版本号采用语义化版本系统，其格式如下：

<major>.<minor>.<patch>

参数说明：

<major>:

主版本号，代表重大更新，通常包括对于新芯片的支持，新功能的引入等

<minor>:

次版本号，代表重要更新，通常包括新命令的添加、bug修复等

<patch>:

修订版本号，表示修复了一些问题，但没有新增任何功能

示例

// send ATCMD
AT+GMR
// receive ATCMD response
+GMR:
ATCMD VERSION: v2.2.1
SDK VERSION: v3.5

编译和下载

准备工作

除了从 GitHub 获取SDK发布版本之外，用户也可以自行使用 {sdk} 编译固件。

关于如何编译SDK，请参考 编译代码。

SSL证书

如果在使用 AT 命令时需要 SSL 证书，用户可通过以下两种方式，在执行AT命令之前准备好证书。将预先准备好的证书文件存储在 VFS 二进制文件中。证书名称的格式和相关的 AT 命令如下：

  • 将预先准备好的证书文件按以下名称和格式存储在 VFS 二进制文件中，然后将它们与应用固件一起烧录到 AT 模块中。

    <client/server><ca/cert/key><1/2/3>.crt

  • 在 AT 模块运行期间，使用 AT+FS 命令将证书写入文件系统。AT+FS 命令的具体用法请参考 文件系统AT命令集。

备注

在创建 VFS 二进制文件时，SSL 证书应放置在 KV 目录中。文件系统 AT 命令的操作也在 KV 目录中进行。

配置

对于 MCU 控制模式，配置通过 atcmd_config.json 文件来实现。

请将 atcmd_config.json 文件转换为 LittleFS二进制文件，并放置在 KV 目录中（参考 :ref:示例 <atcmd_config_example>）。

atcmd_config.json 文件的具体格式如下：

{
   "interface":"spi",

   "spi":
   {
      "mosi":"PA27",
      "miso":"PA28",
      "clk":"PA26",
      "cs":"PA12",
      "master_sync_pin":"PB30",
      "slave_sync_pin":"PB31"
   },

   "uart":
   {
      "baudrate":115200,
      "tx":"PB19",
      "rx":"PB18"
   }
}

用户可以选择通信接口为 UART 或SPI，并修改接口的属性值。UART和SPI模式使用的引脚同样可以定制。

在 MCU 控制模式下，使用SPI通信接口时，如果希望实现更好的高速传输性能，则需要在 menuconfig 中启用 High TP 模式。配置过程如下：

示例

本节演示如何准备自定义的 atcmd_config.json 文件并激活它：

  1. 创建一个新的 atcmd_config.json 文件，并填写以下自定义内容：

    {
   "interface":"uart",

   "uart":
   {
      "baudrate":115200,
      "tx":"PA26",
      "rx":"PA27"
   }
}

  2. 创建一个 TEST/KV 目录，并将 atcmd_config.json 文件放到 KV 目录下。

  3. 使用 LittleFS 二进制文件创建工具生成一个LittleFS 二进制文件（参考：VFS二进制文件生成）。

    linux@user:~$./mklittlefs -b 4096 -p 256 -s 131072 -c TEST lfs.bin
/KV/atcmd_config.json

  4. 根据Flash布局，将生成的 lfs.bin 文件烧录到AT模块的Flash中的VFS1区域 （参考 Flash中的VFS）。

  5. 烧录支持MCU控制模式的应用固件（参考 编译 ）。

  6. AT 模块上电后，如果在 LOGUART 端口上观察到自定义配置的打印输出（如下），表明配置已生效。

    [AT-I] ATCMD MCU Control Mode : UART, tx:PA26, rx:PA27, baudrate:115200

编译

在完成上述准备工作后，用户可以在 {sdk} 目录下编译固件。具体步骤如下：

  1. 运行 make menuconfig 命令，选择 MCU Control 或 LOGUART。

    // Your SDK direction
cd $ {sdk}
// The chip type you choose, e.g. amebadplus_gcc_project
cd source/{ameba_name}
make menuconfig
// ...

  2. 运行 make all 命令重新编译工程。

编译成功后，可以在 {ameba_name} 目录下找到固件。

如果想使用蓝牙AT命令集，先运行 $make menuconfig 命令来启用 BLE Transfer Module。

下载

有以下两种方法可以将固件下载到Flash：

  • Image Tool：Realtek 提供的软件（推荐使用）

  • GDB Server：主要用于 GDB 调试的用户场景

在本节中，我们介绍第一种方法。

Image Tool 是 Realtek 为 Ameba 系列 SoC 开发的固件下载工具，可以通过 UART 接口将固件下载到设备的 Flash 中。

当启动 Image Tool 时，界面显示如下所示：

设备配置文件提供了固件下载所需的必要设备信息，其命名规则如下：

<SoC name>_<OS type>_<Flash type>[_<Extra info>].rdev

对于不同的Ameba芯片来说，在下载固件到Flash之前，点击 File > Open 选择对应的 .rdev 文件，然后选择对应的固件。

在下载固件之前，应先使芯片进入下载模式，步骤如下：

  1. 按住 DOWNLOAD 按钮

  2. 按下 CHIP_EN 按钮

  3. 同时释放以上两个按钮（此时，芯片进入下载模式）

  4. 使用USB线将芯片模块连接到PC，并点击Image Tool中的 DOWNLOAD 按钮，开始下载固件
