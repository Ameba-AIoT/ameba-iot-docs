WiFi Bridge Host 驱动移植指南

当前驱动代码已在 Linux Kernel v5.15 完成验证，其他内核版本若遇编译问题请联系 Realtek 技术团队。

本地SDK环境操作

  1. 进入驱动配置目录:

    cd {SDK}/component/wifi/whc_host_linux

  2. 执行接口部署脚本（支持sdio/spi等接口）:

    ./fullmac_setup.sh sdio bridge

    ./fullmac_setup.sh spi bridge

    备注

    SPI 会在 CS 引脚置低并检测到 CLK 后立即进行数据传输。总线繁忙时可能存在边界条件： 测试表明，在 CS 置低与主机推送 CLK 之间增加 7μs 延迟可确保总线繁忙时的数据传输安全。 但部分旧版本系统的 SPI 驱动可能不支持 spi_delay，此时需直接修改驱动代码以添加延迟。

  3. 在交互界面中选择目标设备型号

  4. 将 whc_host_linux 目录完整拷贝至Linux主机

Linux主机操作

  1. 安装编译工具链:

    sudo apt-get install build-essential

  2. 编译并加载驱动模块:

    cd {driver_path}/whc_host_linux
make
sudo cp bridge/sdio_bridge/bridge_sdio.ko /lib/modules/$(uname -r)/
sudo systemctl stop dhcpcd.service
sudo depmod -a
sudo modprobe bridge_sdio

    cd {driver_path}/whc_host_linux
make
sudo cp bridge/spi_bridge/bridge_spi.ko /lib/modules/$(uname -r)/
sudo systemctl stop dhcpcd.service
sudo depmod -a
sudo modprobe bridge_spi

    备注

    • {driver_path}: 驱动目录实际存储路径

    • $(uname -r): 自动获取当前内核版本

  3. 启用网络接口:

    sudo ifconfig eth_sta0 up
sudo ifconfig eth_sta1 up

  4. 验证接口状态:

    ifconfig

当前驱动代码已在 Raspberry Pi 4 Model B (Broadcom BCM2711) 和 Raspberry 5 (Broadcom BCM2712) 完成硬件验证，它们分别兼容Raspberry Pi OS基于Debian 11和Debian 12的发行版。

  1. 启用SDIO接口配置：

    树莓派4或者4B：

    # Add device tree overlay in /boot/config.txt
sudo sh -c 'echo "dtoverlay=sdio,poll_once=off" >> /boot/config.txt'

    树莓派5：

    # Add device tree overlay in /boot/firmware/config.txt
sudo sh -c 'echo "dtoverlay=sdio,poll_once=off" >> /boot/firmware/config.txt'

备注

此配置将禁用SDIO总线轮询机制以提升传输稳定性

  1. 后续操作与“Linux环境”步骤3-7完全兼容：

  1. 在树莓派中使能SPI接口:

    sudo raspi-config

  2. 选择 Interface Options > SPI > Yes:

  1. 生成设备树overlay:

sudo su
cd {driver_path}/whc_host_linux/common/spi
dtc -@ -Hepapr -I dts -O dtb -o inic_spidev.dtbo spidev-overlay.dts
cp inic_spidev.dtbo /boot/overlays/
dtoverlay inic_spidev

  1. 后续操作与“Linux环境”步骤完全兼容：

whc_bridge_host_buf_rx_to_user

__attribute__((weak)) int whc_bridge_host_buf_rx_to_user(u8 *buf, u16 size);

项目

描述

简介

接收从设备到主机的数据。 这是一个弱函数，SDK 会将数据从内核发送到用户空间。

不使用 bridge app 的客户需要自己实现该函数。

参数

  • buf: 接收到的数据地址

  • size: 接收到的数据长度（包含填充）

返回值

数据接收的结果：

  • 0：数据正常传送到用户空间

  • 其他：数据接收异常

APP 架构

APP的demo通过两类命令实现双向控制：

  1. 内核级配置

    • setmac: 配置设备MAC地址

    • netifon: 激活网络接口载波

  2. 设备级控制

    • getmac: 获取设备MAC地址

    • getip: 获取设备IP配置

    • setrdy: 通知设备主机状态

通信架构

APP 支持的命令

命令格式

功能描述

init

初始化内核Netlink通信通道

getmac <device_idx>
获取指定设备MAC地址

  • device_idx: 设备端口索引，只能是0或1：其中0对应eth_sta0（设备端的STA），1对应eth_sta1（设备端的SoftAP）。

getip <device_idx>
获取设备网络层配置

  • device_idx: 设备端口索引，同“getmac”命令。

setrdy

通知设备端主机准备就绪

setmac <device_idx>   <mac>
配置接口硬件地址

  • device_idx: 设备端口索引，同“getmac”命令。

  • mac: MAC格式 (xx:xx:xx:xx:xx:xx)

netifon <device_idx>
激活接口物理层载波

  • device_idx: 设备端口索引，同“getmac”命令。

备注

  1. 在我们提供的内核驱动中，网络设备 (net_device) 的载波 (carrier) 状态默认是关闭的，因此设备不会正常工作。“netifon”命令用于开启网络设备的载波状态（相当于调用 netif_carrier_on 函数）。客户在开发应用程序时，需要注意调用 netif_carrier_on 函数以确保设备正常运行。

  2. 在我们提供的内核驱动中，默认情况下，网络设备 (net_device) 的物理地址（MAC 地址）被设置为 00:00:00:00:00:00。使用“setmac”命令可以为网络设备配置正确的物理地址。在开发应用程序时，客户需要注意为网络设备设置正确的物理地址。有关此过程的详细信息，请参考“getmac”和“setmac”命令。
