介绍

OTA（Over-The-Air）技术，即空中下载技术，是一种通过无线网络远程更新设备固件的技术‌。通过 OTA 更新，设备通过无线网络接收新的固件来升级软件版本， 而无需物理访问每个设备。这不仅提高了更新效率，降低了维护成本和复杂度，还确保了设备能够及时获得安全补丁和性能优化‌，提升设备的性能和用户体验‌。

随着物联网设备应用场景的日益复杂，我们提出了一种增强的 OTA 解决方案， 除了支持传统的无线更新外，还引入了非无线传输方案。这一增强功能特别适用于网络连接不稳定或不具备无线联接能力的设备。 我们的非无线传输方案可以通过有线连接（如 SPI、UART 串口）进行更新，或者通过预先载有更新内容的存储介质（如 SD 卡、文件系统）实现。 这种方法不仅提供了更大的灵活性，还保证了在多种环境下的设备可用性和数据完整性。

固件标签和版本号

在 OTA1/OTA2 之间的启动选择主要依赖于证书和清单中的固件标签和版本号。如下图所示，证书和清单中包含8字节的固件标签，2字节的主版本号和2字节的次版本号。

固件标签是一个固定的8字节长度的数据。OTA 选择流程检查固件标签的有效性，引导程序和应用程序有不同的固件标签。

版本号是主版本号和次版本号的组合。OTA 选择流程比较版本号的大小，版本号计算方式如下：

\[\text{版本号} = (\text{主版本号} \ll 16) \, | \, \text{次版本号}\]

设备在 OTA 升级后的下一次启动，引导程序中会检查固件标签的有效性和比较版本号大小，以决定从 OTA1 或 OTA2 启动。固件标签和版本一致性确保了可靠的启动。每个固件必须检查以下项目：

  1. 检查 OTA1 和 OTA2 的固件标签是否有效。

    • 如果只有一个固件标签有效，则从有效的固件启动。

    • 如果两个固件标签都无效，则启动失败。

  2. 比较 OTA1 和 OTA2 的版本号。

    • 如果 OTA1 和 OTA2 都有效且版本号不同，设备将从版本号更大的固件启动。

    • 如果 OTA1 和 OTA2 都有效但版本号相同，设备将从 OTA1 启动。

注意

  • 引导程序在清单中检查固件标签和版本号，版本号范围为 0 到 32767。

  • 应用程序在证书中检查固件标签和版本号，版本号范围为 0 到 65535。

防回滚

防回滚功能用于防止版本回滚攻击。当防回滚启用时，证书或清单中的版本号不能小于设置在 OTP 中的防回滚版本。 否则，该固件将被视为无效，而芯片将不会从无效固件启动。 通常，如果 OTA 更新与安全性相关，用户可以在 OTP 中编程一个更大的防回滚版本号，并同时更新带有更大主版本的固件以防止回滚攻击。

防回滚流程如下所示。一旦防回滚启用，设备将分别比较从 OTA1 和 OTA2 固件获取的主版本号与 OTP 中的防回滚版本号。 如果固件的主版本号小于防回滚版本号，则该固件将被视为无效。

OTA 压缩固件

当启用 OTA 固件压缩功能时， OTA 固件将被压缩，从而减小固件的大小，这可以有效节省 Flash 空间。

OTA 固件压缩仅适用于后缀为 app.bin 的应用固件。压缩完成后，生成的压缩固件文件将放在目录 image 中，文件后缀为 app_compress.bin 。 同时，压缩后的应用固件将被整合到 ota_all.bin 文件中。

压缩算法采用 LZMA（Lempel-Ziv-Markov 链算法），这是一种由 Igor Pavlov 开发的无损数据压缩算法，以其高压缩比和相对较低的解压内存要求而著称。 LZMA 算法通常用于 7z 格式的文件压缩，并广泛应用于 7-Zip 软件。

在这里，我们使用压缩率来衡量压缩效率。压缩率的计算公式如下：

\[\text{压缩率（％）} = \left( \frac{\text{压缩后固件大小}}{\text{原始固件大小}} \right) \times 100\]

压缩率会受到固件内容的影响。通常，压缩率在 60 ％到 70 ％之间。原始固件内容越复杂，压缩率往往越高。

一旦压缩的应用固件被下载到一个 OTA 分区，系统将始终优先选择压缩固件。压缩固件将解压到另一个 OTA 分区，然后系统从该分区启动。 这意味着两个分区之间只有一个有效的应用固件。例如，如果设备目前从 OTA1 启动并运行 OTA 应用，压缩固件将被下载到 OTA2。 下一次启动时，系统将选择压缩固件，将其解压到 OTA1 ，然后继续从 OTA1 启动。

用户可通过以下步骤生成 OTA 压缩固件：

  1. 进入项目并打开配置菜单。

    cd project folder
./menuconfig.py

  2. 选择启用压缩固件，保存并退出。

    *** --------MENUCONFIG FOR General--------- ***
CONFIG TrustZone  --->
...
CONFIG OTA OPTION  --->
   [ ] Enable Bootloader OTA
   [*] Enable Compress APP Image

  3. 通过以下命令编译项目，包含压缩应用固件的 ota_all.bin 将生成在 {SDK}\amebadxxx_gcc_project 中。

    cd project folder
./build.py

修改配置

  1. 更新项目文件夹中配置文件 manifest.json 中的版本号。

    1. 为 KM4 引导程序配置主版本和次版本。

      "boot":

{

   ...
},

  2. 调整引导程序的防回滚版本号，并在必要时启用防回滚。

    1. 更改引导程序的防回滚版本号

      默认情况下，所有固件使用 OTP 中相同的防回滚版本作为阈值以防止防回滚攻击。

      名称

      OTP 地址

      长度

      描述

      引导程序版本号

      物理地址 0x36E~0x36F

      16 位

      引导程序的防回滚版本号

      防回滚的引导程序版本默认值为0。用户可以更改’0’位的数量以增加引导程序版本。

      例如，用户可以通过以下命令将防回滚的引导程序版本设置为 1 ：

      EFUSE wraw 36E 2 FFFE

    2. 启用防回滚

      用户可以通过以下命令写入 OTP 以启用防回滚。

      EFUSE wraw 368 1 BF

    注意

    • 一旦启用防回滚，就无法禁用。

    • 如果引导程序和应用程序不使用相同的防回滚版本，修改 {SDK}\component\soc\amebaxxx\bootloader\boot_ota_km4.c 或 {SDK}\component\soc\amebaxxx\bootloader\boot_ota_hp.c 中的 BOOT_OTA_GetCertRollbackVer() 并在 OTP 中为应用程序定义另一个防回滚版本。

  3. 如果用户需要升级引导程序，则将引导程序的 OTA2 地址写入 OTP。根据 Flash_Layout 设置此地址，参阅 用户配置 。

    EFUSE wraw 36C 2 0082

    EFUSE wraw 36C 2 0082

    EFUSE wraw 36C 2 0083

    注意

    • 引导程序 OTA2 地址为 OTP 地址 0x36C 中的值左移 12 位的值，或 OTP 地址 0x36C 中的值 * 4K。

    • 如果引导程序 OTA2 的默认地址为 0xFFFFFFFF ，则在 OTA 期间不会进行引导程序升级，设备总是从引导程序 OTA1 启动。

    • 在串行终端工具中使用以上命令。

  4. 重新编译项目。

  5. 将固件下载到 Flash ，然后重置设备。

  • OTA_IMGID_BOOT: 0x0

  • OTA_IMGID_APP: 0x1

OTA 流程

固件升级按以下步骤实现：

  1. 连接到服务器。通过网络进行的 OTA 需要设置 IP 地址、端口和 OTA 类型。

  2. 确定要升级的目标分区。如果当前运行分区是 OTA1 ，则 OTA2 将被选择进行升级。否则，将选择 OTA1。

  3. 接收 OTA 头以获取目标升级固件的信息，如上表所示。

  4. 下载新固件。

  5. 擦除新固件的 Flash 空间，并将其写入 Flash ，但不包括清单数据。

  6. 验证校验和：

    • 如果计算出的校验和与从 OTA 头获得的校验和不匹配，则 OTA 流程失败。

    • 如果计算出的校验和与从 OTA 头获得的校验和匹配，则将清单数据写入升级区域，表示设备应从新固件启动。

  7. OTA 完成。重置设备，设备将从新固件启动。

注意

上述提到的清单数据包括固件标签和版本号。固件标签和版本一致性确保了可靠的启动。

OTA 例程

按照以下步骤运行 OTA 例程以从 HTTP 服务器升级：

  1. 编辑 example_ota.c。

    1. 根据服务器IP地址编辑主机。

      #define PORT   8082
static const char *host = "192.168.31.193";   //"m-apps.oss-cn-shenzhen.aliyuncs.com"
static const char *resource = "ota_all.bin"; //"051103061600.bin"

    2. 将 OTA 类型编辑为 OTA_HTTP。

      ret = ota_update_init(ctx, (char *)host, PORT, (char *)resource, OTA_HTTP);

  1. 使用命令 ./build.py -a ota 重新编译项目并将固件下载到设备。

  2. 按照 固件标签和版本号 描述修改 manifest.json 中的主版本号和次版本号为更大的版本。

    注意

    引导程序默认选择带有更大版本号的 OTA 固件。 如果用户不想修改版本号，请在步骤 2 之前在 ameba_ota.h 中定义的 OTA_CLEAR_PATTERN 修改为1。 这只应在开发阶段使用。

  3. 重新编译项目，并将 ota_all.bin 复制到 DownloadServer(HTTP)。

  4. 编辑 http_server.py。

    server_ip = '192.168.31.193'
server_port = 8082

  5. 使用命令 python http_server.py 执行脚本，并启动下载服务器程序。

  6. 重置设备并连接到 HTTP 服务器。

  7. 固件下载完成后，设备将自动重置并从新固件启动。
