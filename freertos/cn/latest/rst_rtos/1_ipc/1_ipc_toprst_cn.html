IPC 架构

  1. IPC定义

    • 处理器间通信（Inter-Processor Communication, IPC）是专为多核异构系统设计的硬件加速通信机制，通过共享SRAM实现跨核数据交互，确保多核间低延迟的信息传递。其核心目标是为多核之间提供高效协同的工作能力。

  2. 硬件架构特性

    1. 对称结构

      • 每个CPU都有一套相同的IPC模块。

    2. 双工通信模型

      • 支持双向全双工通信。

    3. 通道隔离性

      • 通道间采用硬件级资源隔离设计，任意通道的数据传输互不阻塞。

      • 支持并行多通道操作（如CPUA可同时通过通道n发送传感器数据、通道m传输日志信息）。

    4. 信息池

      • 预分配的一块内存，每个IPC通道通过信息池中预设的固定地址来收发信息。

    5. 共享内存

      • 当收发消息大于4字节时，发送端动态分配一块共享内存，并将内存地址作为消息写入信息池，接收端在发送端释放共享内存前，将数据拷贝到本地。

      • 当收发消息小于等于4字节时，将消息写入信息池。

    6. NVIC中断生成器

      • 在IPC通道准备好数据之后，通过写IPC某块中的寄存器，在接收端产生接收中断， 同样在接收端完成接收后，也可以在接收端触发发送完成中断。

  1. 数据流模型

    • 发送端流程：

      [共享信息] → 选择可用通道 → 写入共享内存 → 更新通道寄存器 → 触发对端中断 → 对端读取内存

    • 接收端流程：

      中断上报 → 读取通道寄存器 → 定位共享内存 → 拷贝数据到本地 → 处理数据 → 清除中断标志

    • 关键寄存器 (KR4侧为例)

      • 两组传输信道

        • REG_TX_DATA_R2M

        • REG_RX_DATA_M2R

      • 中断状态寄存器

        • REG_ISR_KR4

      • 中断掩码寄存器

        • REG_IMR_KR4

      • 中断清除寄存器

        • REG_ICR_KR4

    • 端到端触发机制 （KR4发送至km4为例）

      • REG_TX_DATA_R2M用于KR4向KM4传输数据，一旦将位“1”写入通道n（n=0，1，…，7），目标REG_RX_DATA_R2M中的n位将被设置。REG_ISR_KM4中通道n的Rx full将同时设置。

      • 将位“1”写入REG_ISR_KM4中的通道n会清除通道n的Rx full中断，REG_ISR_KR4中相应的通道n的Tx empty中断位将自动设置, 同时REG_TX_DATA_R2M和REG_RX_DATA_R2M的通道n位也会被清除。

IPC 配置方法

  1. 通道选择与定义

    • 概念：IPC通道是连接两个IPC实体之间的双向独立通信路径，每个方向包含多组通道。

    • 作用：用于在两个CPU之间传输发送和接收事件，实现核间通信。

    • 文件路径 : ameba_ipccfg.h

    • 选择与定义示例：

      在 ameba_ipccfg.h 文件中，可以通过取消注释并修改通道定义来选择和自定义通道。例如：

      1/* 取消通道n的注释，并添加功能描述 */
2#define IPC_N2A_Channeln  n  /*!< 用户自定义通道：KM0到KM4的传感器数据传输 */

    • 注意事项： 避免使用系统已预留的通道，这些通道可能用于WiFi/BT等系统组件。

  2. 组件概念与定义

    1. IPC配置表（IPC_INIT_TABLE）

      • 作用：定义了整个传输链路的所有信息，包括数据类型、接收和发送回调函数、中断处理函数参数、传输方向和IPC通道号。

      • 使用说明：用户需要将此结构体实例存放到 IPC_TABLE_DATA_SECTION 中，SDK会根据结构体的成员信息开启相应通道的Tx/Rx中断，并注册中断处理函数。

      • 结构体定义：

        1typedef struct _IPC_INIT_TABLE_ {
2   u32 USER_MSG_TYPE;  /* 数据类型：IPC_USER_DATA（小于32bit数据）或IPC_USER_POINT（大于32bit数据的地址） */
3   void (*Rxfunc)(void *Data, u32 IrqStatus, u32 ChanNum); /* 接收中断回调函数 */
4   void *RxIrqData;    /* 接收回调函数参数 */
5   void (*Txfunc)(void *Data, u32 IrqStatus, u32 ChanNum); /* 发送完成中断回调函数 */
6   void *TxIrqData;    /* 发送回调函数参数 */
7   u32 IPC_Direction;  /* 传输方向 */
8   u32 IPC_Channel;    /* IPC通道号 */
9} IPC_INIT_TABLE, *PIPC_INIT_TABLE;

    2. IPC数据结构（IPC_MSG_STRUCT）

      • 作用：在消息池中，每个IPC通道拥有一个固定位置用于存放 IPC_MSG_STRUCT 结构体，用于存储和传输数据。

      • 结构体定义：

        1typedef struct ipc_msg_struct {
2   u32 msg_type;   /* 数据类型，与USER_MSG_TYPE相同 */
3   u32 msg;        /* 需要传递的32bit数据或数据地址（对于大于32bit的数据） */
4   u32 msg_len;    /* 数据长度 */
5   u32 rsvd;       /* 保留字段 */
6} IPC_MSG_STRUCT, *PIPC_MSG_STRUCT, ipc_msg_struct_t;

  3. 通信协议栈注册

    • 中断回调注册（KM4侧）

      • 在KM4中为从KM0到KM4的通道n添加一个新的 IPC_INIT_TABLE。

       1/* 通道配置表*/
 2IPC_TABLE_DATA_SECTION
 3const IPC_INIT_TABLE ipc_chn_config = {
 4   .msg_type    = IPC_USER_POINT,     // 消息传递方式：指针传递
 5   .rx_func     = KM4_Channeln_Handler,
 6   .rx_irq_data = NULL,               // 无附加参数
 7   .tx_func     = IPC_TXHandler,      // 发送完成回调（SDK标准API）
 8   .tx_irq_data = NULL,
 9   .direction   = IPC_KM0_TO_KM4,     // 方向标识
10   .channel_num = IPC_N2A_Channeln    // 通道号宏定义
11};

      • SDK将自动启用IPC中断，并为该通道注册相应的IRQ处理程序和数据。在本例中，当KM0通过通道n发送IPC时，KM4将进入 KM4_Channeln_Handler() （即通道n的中断处理函数）。SDK会在IRQ退出后清除对应通道的ISR。

      1/* KM4接收处理函数（中断上下文）*/
2void KM4_Channeln_Handler(void *data) {
3   PIPC_MSG_STRUCT msg = ipc_get_message(IPC_KM0_TO_KM4, IPC_N2A_Channeln);
4   uint32_t *sensor_data = (uint32_t*)msg->msg;  // 提取共享内存数据
5   _memcpy(data, (void*)sensor_data, sizeof(data));  // 内存屏障保护
6   process_sensor_data(data);            // 用户数据处理
7}
    • 发送端配置（KM0侧） （可选）

      • 如果有多笔数据需要发送，为了避免轮询发送状态占用cpu资源, 可以给tx 的通道n注册IRQ处理程序。我们建议将此处理函数注册为 ameba_ipc_api.c 中的 IPC_TXHandler() 。在此函数中，如果使能了 config_kernel ，则会释放通道n的信号量（tx sema），在 ipc_send_message() 中会去获取信号量，以便及时启用下一次传输并避免任务阻塞。SDK会在IRQ退出后清除对应通道的ISR。在这种情况下，需要在KM0中为从KM0到KM4的通道n添加一个新的 IPC_INIT_TABLE。

       1/* KM0发送完成回调*/
 2void IPC_TXHandler(uint32_t status) {
 3
 4   IPC_INTConfig(IPCx, ChanNum, DISABLE); // 关闭中断
 5   ...
 6   rtos_sema_give(tx_sema);  // 释放信号量继续发送
 7   ...
 8
 9}
10
11/* KM0发送配置表 */
12IPC_TABLE_DATA_SECTION
13const IPC_INIT_TABLE ipc_km0_tx_config = {
14   .msg_type    = IPC_USER_POINT,
15   .rx_func     = NULL,              // 接收函数置空
16   .tx_func     = IPC_TXHandler,    // 发送完成回调（SDK标准API），可自定义
17   .tx_irq_data = NULL,
18   .direction   = IPC_KM0_TO_KM4,     // 方向标识
19   .channel_num = IPC_N2A_Channeln    // 通道号宏定义
20};

  1. 实现数据收发

    • KM0发送数据

      • 当KM0通过通道n向KM4发送IPC请求时，应调用 ipc_send_message() 函数，并指定通道号和消息。如果不需要消息，则只需将 ipc_send_message() 的第三个参数输入为NULL。

       1void send_sensor_data(void) {
 2
 3   IPC_MSG_STRUCT ipc_msg_temp;
 4   sensor_buffer = read_sensor();  // 更新共享内存
 5
 6   ipc_msg_temp.msg_type = IPC_USER_POINT; // 传递指针
 7   ipc_msg_temp.msg = (u32)&sensor_buffer;
 8   ipc_msg_temp.msg_len = 1;              // 数据长度（字节）
 9   ipc_msg_temp.rsvd =0;
10
11   /* 发送共享数据(SDK标准API)， 带超时处理 */
12   ipc_send_message(IPC_KM0_TO_KM4, IPC_N2A_Channeln, & ipc_msg_temp);
13}
    • KM4接收处理（中断+任务协作）

      • 在KM4通过KM0的通道n接收到IPC后，它将进入IPC中断处理程序，并执行相应的接收IRQ处理程序。如果需要获取消息，可以调用 ipc_get_message() 函数。

       1/* 中断上下文：快速捕获数据 */
 2void KM4_Channeln_Handler(void *data) {
 3
 4   // get message
 5
 6   rtos_sema_give(ipc_sema);  // 触发任务级处理
 7}
 8
 9/* 任务上下文：数据处理 */
10void ipc_task(void *param) {
11   while (1) {
12      rtos_sema_take(ipc_sema, RTOS_WAIT_FOREVER);

IPC 应用层使用说明

Introduction

IPC queue is designed for cores communication. Define an IPC queue between every two cores, and the two cores can send and receive messages through this IPC queue.

IPC Message Queue:

  1. Packets send data from Core A to Core B through IPC message queue.

  2. Sender copy a complete packet into queue.

  3. Receiver copy packet with custom size from queue.

For receiver, IPC interrupt is necessary.

How to Use IPC Application

Before use IPC application, users should enable this function and initial it in each core.

  • For KM4: make menuconfig > MENUCONFIG FOR KM4 CONFIG > CONFIG IPC Message Queue > Enable IPC Message Queue

  • For KR4: make menuconfig > MENUCONFIG FOR KR4 CONFIG > CONFIG IPC Message Queue > Enable IPC Message Queue

  • For DSP: Open project_dsp with Xplorer, then click project_dsp > Build Properties > Symbols, and set CONFIG_RPC_EN to 1

The procedure will call ipc_app_init() in main() to initial IPC application when IPC application is enabled. The initialization include IPC queue initialization.

How to Use IPC Queue

For example, communication between KM4 and KR4 through IPC_ID_KR4_TO_KM4.

  1. IPC_ID_KR4_TO_KM4 definition in enumeration IPC_ID in ipc.h is shown below. This definition is used in Stap 2.

    typedef enum {
   IPC_ID_KR4_TO_KM4 = 0,
   IPC_ID_KM4_TO_KR4 = 1,
   IPC_ID_KR4_TO_DSP = 2,
   IPC_ID_DSP_TO_KR4 = 3,
   IPC_ID_DSP_TO_KM4 = 4,
   IPC_ID_KM4_TO_DSP = 5,
   /*14-19 used for RPC*/
   IPC_ID_NUM        = 20
} IPC_ID;

  1. When KR4 sends a message to KM4 through IPC_ID_KR4_TO_KM4, it should call IPC_Message_Queue_Send() and specify the queue ID and message.

    int32_t ret =  IPC_Message_Queue_Send(IPC_ID_KR4_TO_KM4, buf, IMQ_BUF_SIZE, WAIT_FOREVER);

  2. When KM4 receives a message from KR4 through IPC_ID_KR4_TO_KM4, it should call IPC_Message_Queue_Recv() and the queue ID and a buffer to receive message.

    size = IMQ_BUF_SIZE;
int32_t ret =  IPC_Message_Queue_Recv(IPC_ID_KR4_TO_KM4, buf, &size);

备注

  • The memory regions are only defined in AP core, the maximum data region is 40kB.

  • Users can use default test demo to verify IPC queue by command test_ipc_message in console both in KR4 and KM4.

Introduction

IPC queue is designed for cores communication. Define an IPC queue between every two cores, and the two cores can send and receive messages through this IPC queue.

IPC Message Queue:

  1. Packets send data from Core A to Core B through IPC message queue.

  2. Sender copy a complete packet into queue.

  3. Receiver copy packet with custom size from queue.

For receiver, IPC interrupt is necessary.

How to Use IPC Application

Before use IPC application, users should enable this function and initial it in each core.

  • For KM4: make menuconfig > MENUCONFIG FOR KM4 CONFIG > CONFIG IPC Message Queue > Enable IPC Message Queue

  • For KR4: make menuconfig > MENUCONFIG FOR KR4 CONFIG > CONFIG IPC Message Queue > Enable IPC Message Queue
